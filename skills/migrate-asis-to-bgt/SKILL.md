# ASIS → BGT 화면 마이그레이션

> **실행 모델**: Phase 0~2(해석·분석·계약 초안)는 인터랙티브, **Phase 2.5(grill-me 합의 게이트)가 마지막 상호작용**, Phase 3~8(구현·검증·보고서)은 "무개입 실행 규칙"으로 질문 없이 완주한다.
> 주의: team-bgt 플러그인에 이 스킬의 사본이 있다 — 이 로컬 문서가 원본이며 사본 동기화는 별도 관리(드리프트 주의).

## 파라미터

| 파라미터 | 설명 | 예시 |
|---------|------|------|
| `asis_menu_path` | ASIS PMS 메뉴 path 또는 화면 코드 (첫 번째 파라미터) | `PMS > 예산/내역 > 도급 관리 > 도급내역 집계표`, `MPMC10001` |
| `bgt_path` | BGT 대상 경로 | `bgt-fe/src/pages/contract` |

둘 중 하나라도 없으면 작업 시작 전에 반드시 요청한다. 메뉴 path는 PMS 화면을 기준으로 한다.
초기 입력에 구현 범위·유형 판별·stub 정책 등 결정 사항을 함께 주면 Phase 2.5(grill-me) 안건에서 "기확정"으로 제외된다.

## 모델 선택

각 단계의 서브에이전트를 디스패치할 때 아래 모델로 실행하고, 프롬프트에 해당 지시를 함께 준다.

- **분석 단계** — `model: opus`. 프롬프트에 "소스 근거로 빠짐없이 깊게 분석하라(가정 금지)"를 명시한다.
- **구현 단계** — `model: sonnet`. 프롬프트에 "BGT 표준·ASIS 1:1 구성을 지켜 꼼꼼히 구현하라"를 명시한다.
- **검증 단계** — `model: opus`. 프롬프트에 "기준(ASIS/개발 표준) 대비 엄격하고 철저하게 검증하라"를 명시한다.

## Compact 전략 (컨텍스트 관리)

Phase 0~8 장기 오케스트레이션이라 중간에 컨텍스트가 compaction(요약)될 수 있다. 요약 경계를 넘어 안전하게 재개하려면 **디스크 산출물을 상태의 단일 출처(source of truth)로 삼는다.**

> **우선순위**: 하니스 기본 compact 전략이 있더라도, 이 스킬 실행 중에는 아래 compact 전략을 우선 적용한다.

- **무거운 읽기는 서브에이전트에 가둔다.** ASIS 분석·`docs/sp`·`cst` 템플릿 등 대량 읽기는 분석/구현 서브에이전트(`bgt-fe-implementer`·`bgt-be-implementer`) 격리 컨텍스트에서 수행하고, 메인에는 압축 요약 + API 계약(엔드포인트·VO camelCase 키·SP alias↔키 맵)만 반환한다. 메인 컨텍스트에 원본 파일을 덤프하지 않는다.
- **진행 원장(`docs/migration/{화면명}/_progress.md`)을 각 Phase 종료 시 갱신한다.** 기록 항목: (a) 현재 Phase, (b) 확정된 API 계약, (c) 생성/변경 파일 목록, (d) 미해결 불일치 표(Phase 4~6), (e) 분석 문서·보고서 파일 경로, (f) **결정 로그**(Phase 2.5 형식 — 재개·자동 결정의 단일 근거).
- **compaction 후 재개**: 먼저 `_progress.md` + 분석 문서를 다시 읽어 현재 Phase·API 계약·미해결 항목을 복원하고, **이미 통과한 Phase는 재실행하지 않는다.**

## 사전 조건: com 서버 확인

이 스킬은 com 호스트 서버(dev `http://localhost:3000/com/`, bgt remote `:3009` vite dev)가 이미 기동 중이라고 가정한다. **스킬 시작 시 가장 먼저 com 서버 가동 여부를 확인**하고, 그 결과로 Phase 3 구현 검증 방식을 정한다.
- **가동 중(기본)**: Phase 4에서 Playwright로 실제 화면을 열어 기능·UI를 라이브 검증할 수 있으므로, Phase 3 FE 구현에 별도 TDD(vitest)는 필수가 아니다.
- **미가동/접속 불가**: 사용자에게 알린다(시작 시점 = 인터랙티브 구간). 라이브 검증 불가 시의 FE 대체 검증 방식(TDD 여부, 아래 Phase 3 참고)은 Phase 2.5 안건으로 편입해 확정한다.

## 워크플로우

### Phase 0: 화면 경로 해석 (pms-screen-finder)

`asis_menu_path`(메뉴 path 또는 화면 코드)를 받아 `/pms-screen-finder` 스킬로 해당 화면 패키지의 로컬 경로를 찾는다. 이 결과 경로를 이후 단계에서 `asis_path`로 사용한다.

- **ASIS 화면을 검색할 때는 화면 레지스트리 파일 `C:\workspace\bgt\docs\ASIS_메뉴.md`를 1순위 출처로 참고한다.** 이 파일은 ASIS 전체 화면을 `SCREEN_NM`(화면명)·`SCREEN_ID`(화면 코드)·`PSCREEN_ID`(상위 화면, 메뉴 트리 재구성)·`SCREEN_URL`(웹 화면 경로)로 등록한 표다(MPOST WinForms 포함). 메뉴 path/화면명/코드 어떤 입력이든 이 파일에서 먼저 화면 코드·URL을 확정한 뒤 PMS 소스에서 경로를 도출한다. (pms-screen-finder의 Step 0이 이 조회를 수행하므로 통상 그 결과를 그대로 받으면 된다.)
- pms-screen-finder가 반환한 패키지 경로·대상 파일(leaf)·구성요소 맵을 Phase 1 분석의 입력으로 삼는다.
- **screenid = 소스 폴더명**: ASIS_메뉴.md에서 확정한 `SCREEN_ID`는 해당 화면 소스가 든 폴더명과 일치한다. 그 폴더의 `*.designer.cs`(WinForms Designer 생성 파일)를 찾아 AS-IS UI의 기준으로 확보한다 — Phase 1 분석·Phase 3 UI 구현·Phase 4 ① 검증이 공통으로 이 파일을 AS-IS UI 원본으로 참고한다.
- 패키지를 못 찾거나 후보가 여럿이면 결과를 사용자에게 제시해 확정한 뒤 진행한다.

### Phase 1: ASIS 분석

`asis_path`(Phase 0에서 해석한 화면 패키지 경로)를 **화면 영역별 구조 분석 패스**로 분석한다. (이 분석 방식은 understand-anything 스킬의 코드베이스 분석 접근에서 착안한 것이며, 해당 스킬을 설치·호출할 필요는 없다 — 아래 기술된 방법만 그대로 적용한다. pnpm·python·git 의존 없음.)

**화면 영역별 분석 서브에이전트 fan-out.** 화면을 영역 단위(목록 화면 / 상세 화면 / 공통 팝업)로 나눠, 각 영역마다 read-only 분석 서브에이전트를 병렬로 디스패치한다. 해당 영역이 없으면(예: 팝업 없는 화면) 그 에이전트는 생략한다. 각 서브에이전트는 자기 영역에 대해 다음 순서로 구조를 파악하고 결과를 보고한다.
- 파일 인벤토리: 영역에 속한 컴포넌트·모달·라우팅 파일 식별
- 컴포넌트 간 관계: 호출·데이터 흐름·상태 관리 연결
- 백엔드 연동: API 호출, 프로시저명, 파라미터
- 권한 처리, 그리드/폼/모달 패턴
- 사용자 관점 기능·폼 항목·하위 그리드·버튼/액션·관련 소스 파일

메인 에이전트는 각 서브에이전트 보고를 모으고, 화면 개요와 탭별 기능 차이를 직접 식별해 합성한다. 화면 규모가 작아 영역 분할이 무의미하면 fan-out 없이 메인 에이전트가 위 방법으로 직접 분석한다.

**ASIS 기능 분석 문서를 작성한다.** ASIS 화면에서 사용자가 수행할 수 있는 기능(화면에서 발생할 수 있는 동작)을 빠짐없이 분석해 문서로 뽑는다.
- 저장 위치: 프로젝트 루트 `docs/migration/{tobe화면명}/` 폴더 (없으면 생성). 이 폴더에 분석 보고서와 마이그레이션 보고서를 함께 저장한다.
- **ASIS UI 기준 확보**: Phase 0에서 확정한 screenid 폴더의 `*.designer.cs`를 AS-IS UI 원본으로 확보한다(Phase 4 ①의 기준). ASIS 화면이 실제 접근 가능하면 화면 스크린샷도 같은 폴더에 저장해 Phase 4 ②(비전)의 기준으로 함께 쓴다.
- 파일명: `{tobe화면명}_analysis_{날짜}.md`
  - `tobe화면명` = `bgt_path`의 대상 화면명, `날짜` = `YYYYMMDD`
  - 예: `cntrct-bid-lst-dtl_analysis_20260619.md`
- 문서는 **사용자 관점 기능 카탈로그**로 구성한다(가정 금지, 소스 근거로만 작성).
  - 화면 개요: 화면명·코드, 목록/상세/팝업 구성, 탭 구성
  - 목록 화면: 탭 전환, 조회 조건(탭별 차이 포함), 조회 실행, 결과 그리드, 신규/액션 버튼, 출력(인쇄·Export), 기타
  - 상세 화면: 모드(신규/수정/재공고 등), 입력 폼 항목, 하위 그리드(개찰·구성사·예가·첨부 등), 저장/삭제, 인쇄/이메일, 업체관리 등
  - 공통 팝업, 탭별 기능 차이 요약, 관련 소스 파일
  - 호출 SP·파라미터 등 기술 매핑은 기본 제외하고, 필요 시 문서 끝에 별도 절로 분리한다.
- 이 문서는 이후 단계의 "ASIS 기능 보존" 기준(추가/삭제 금지 체크리스트)으로 사용한다.
- 이후 구현(Phase 3 병렬 구현)은 이 분석 문서의 내용을 토대로 진행한다.

분석 문서 작성 후 **모든 의심스러운 부분**(기술적 불명확점, 기능 포함 여부, UX 판단 항목)을 **미결 안건 목록**으로 `_progress.md`에 축적한다 — 이 시점에 사용자에게 직접 질문하지 않는다. 안건은 Phase 2.5(grill-me 합의 게이트)에서 일괄 해소한다.

### Phase 2: 참고 화면 탐색 + API 계약 정의 (동시 수행)

**참고 화면**: `bgt-fe/src/pages`, `cst-fe/src/pages`에서 다음 기준으로 자동 탐색한다. 가장 유사한 화면을 구현 참고로 사용한다. 사용자에게 별도로 제시하지 않는다.
- 전체 UI 구조가 유사한 화면 (예: 그리드+상세 패널, 탭+모달, 목록+폼 등 레이아웃 패턴 일치)
- 구현 화면에 검색 영역이 있다면: 검색 항목 수와 구조(드롭다운/입력/날짜 조합 등)가 유사한 화면, 그리고 동일한 라벨명을 사용하는 화면을 추가로 탐색해 검색 영역 참고로 사용
- 절대 임의로 생성하지 말고 bgt-fe / cst-fe / com-fe 순으로 비슷한 UI 화면을 참고해서 만든다. 참고할 화면이 없는 경우는 비워둔 뒤 사용자에게 보고한다

검색 영역 구현 시 아래 정렬 규칙을 반드시 따른다.
- 같은 열(column)의 라벨은 모두 끝나는 지점(우측 끝)이 동일하게 정렬된다.
- 같은 열의 입력 영역(input, select 등)은 너비가 동일하다.

**API 계약**: ASIS 분석에서 파악한 호출을 BGT URL 규칙(`/v1/{L1}/{L2}/{L3}`)으로 변환하고 Method, Request/Response 구조를 정의해 `docs/migration/{화면명}/_api_contract.md`로 저장한다. 별도 승인 게이트 없이 **Phase 2.5(grill-me) 안건에 편입**한다 — 계약 확정은 grill-me에서 이뤄진다.

### Phase 2.5: grill-me 합의 게이트 (마지막 인터랙션)

Phase 0~2는 인터랙티브(경로 후보 확정 등 질문 허용)지만, **이 게이트가 사용자와의 마지막 상호작용**이다.

1. `/grill-me` 스킬을 호출해 다음 안건을 결정 트리가 해소될 때까지 인터뷰한다:
   ① Phase 1에서 축적한 분석 모호점 ② API 계약 초안 ③ 구현 범위(팝업·우클릭·부가기능) ④ 유형 판별·stub 정책(SP 미배포 시) ⑤ BE 작업 포함 여부 ⑥ 워크트리/브랜치 구성 ⑦ 기타 실행 정책 편차. 초기 인자로 이미 확정된 항목은 "기확정"으로 제외.
2. 모든 결정을 `_progress.md`의 **결정 로그**에 기록한다(아래 형식). 이 로그가 무개입 구간의 단일 판단 근거다.
3. **BE 인계(조건부)**: BE 변경이 범위에 있으면 마지막 질문 1개로 기존 BE(IDE/터미널) 종료를 요청하고, 확인 후 `scripts/be-start.ps1`(이 스킬 폴더)로 루프 소유 콘솔에서 BE를 기동해 :7078 헬스를 확인한다. FE-only 실행이면 생략.

**결정 로그 형식** (`_progress.md` 표준 섹션):

| # | 결정 | 근거 | 출처 |
|---|------|------|------|
| D1 | 팝업 9종 전부 구현 | 사용자 지시 | grill-me |

- 출처: `초기입력` / `grill-me` / `자동(결정로그)` / `자동(표준)` / `자동(ASIS)`. 무개입 구간의 자동 결정도 여기에 추가 기록한다.

## 무개입 실행 규칙 (Phase 3~8)

Phase 2.5 통과 후에는 **사용자에게 질문하지 않는다.** 아래 규칙으로 스스로 결정하고 계속 진행한다.

- **모호점 판단 우선순위**: 결정 로그 → BGT 개발표준(`.claude/rules`) → ASIS 동작. 선택과 근거를 결정 로그에 `자동(*)` 출처로 추가하고, 자동 결정 전체를 보고서 §6 "확인 필요 항목"에 목록화한다.
- **BE 재기동**: 사용자에게 요청하지 않는다 — `scripts/be-restart.ps1`을 실행한다(아래 런북). 스크립트가 `NOT_OWNED`/`STOP_TIMEOUT`(정지 실패) 또는 `START_FAILED reason=ORA-xxxxx`(DB 거부)/`START_TIMEOUT`(기동 실패)으로 끝나면 해당 라이브 검증 항목을 BLOCKED 마킹하고 정적 검증으로 폴백해 계속한다. 특히 **`ORA-`(예 ORA-18958)는 재시도하지 않는다** — DB 세션 장애라 재시도로 살아나지 않고 시간이 지나야 회복된다. **어떤 경우에도 java 프로세스를 강제종료하지 않는다**(D-18 §1, ORA-18958).
- **쓰기성 검증 차단**(D-18 §2): 저장/삭제 POST가 auto-mode 분류기에 차단되면 우회하지 않는다 — payload 클라이언트 재현+읽기 간접검증으로 대체하고 계속 진행하며, 실제 POST 확인은 보고서 "사용자 조치" 절로 분리한다.
- **블로커 조기 분류**(D-18 §5): 원인이 수정 범위(`bgt-fe`/`bgt-be`) 밖(배포 SP·인프라·권한)이면 즉시 BLOCKED 마킹 + 진단 기록 후 다음 항목으로.
- **완주 정책**: 커밋하지 않는다(사용자가 `/commit` 별도 실행). 보고서 생성으로 종료한다.
- **재개**: 세션 한도·중단 시 `_progress.md`(결정 로그 포함)를 읽고 통과한 Phase를 재실행하지 않고 이어간다. 유일한 잔여 수동 개입은 사용자의 "이어서" 한마디다.

## BE 재기동 런북 (application.yml 무수정)

번들 스크립트: 이 스킬 폴더의 `scripts/`.

- `be-start.ps1` — 새 콘솔 창에서 `gradlew.bat bootRun` 기동 후 :7078 리슨 폴링(기본 300초). 창이 데스크톱에 남아 사용자가 언제든 직접 Ctrl+C 가능. bootRun 출력을 `bgt-be/build/bootrun.log`로 캡쳐해 폴링 중 **DB 거부(`ORA-xxxxx`)·빌드 실패를 즉시 감지**한다(십수 초 만에 죽어도 300초 헛대기하지 않는다). 출력: `STARTED pid=<pid>`(성공) / `PORT_BUSY pid=<pid>` / `SPAWN_FAIL rv=<n>` / **`START_FAILED reason=<ORA-xxxxx|BUILD_FAILED>`**(fast-fail) / `START_TIMEOUT`(최종 폴백).
- `be-restart.ps1` — :7078 소유 프로세스 확인 → **IDE 디버그 세션(`-agentlib:jdwp`/`debugger-agent.jar`) 또는 콘솔 부재면 `NOT_OWNED` 실패**(인계 안 된 프로세스 보호) → AttachConsole+GenerateConsoleCtrlEvent로 Ctrl+C(SIGINT) 주입 graceful 정지 → 포트 해제 대기(60초, 실패 시 `STOP_TIMEOUT`) → `be-start.ps1` 재기동. **강제종료 폴백 없음.**

BE 수정은 가능하면 모아서 재기동 횟수를 최소화한다(D-18 §4).

- **기동 실패 진단**: `be-start.ps1`은 이제 실패 원인을 직접 반환한다.
  - **`START_FAILED reason=ORA-18958`(또는 다른 `ORA-`)** → **세션 잔존 DB 장애(D-18 §3)**. bootRun 이 :7078을 잠깐 바인딩했다가 Hikari 풀 초기화에서 DB에 거부당해 죽은 상태다 — **코드 문제가 아니다**. 즉시 BLOCKED 마킹, **재시도·강제종료 금지**(재시도해도 DB가 살아나지 않고, 강제종료는 ORA-18958을 악화시킨다). 해당 라이브 검증 항목을 정적 검증으로 폴백하고, DB 세션 회복(시간 경과) 또는 사용자/DBA 조치를 보고서 "사용자 조치" 절로 분리한다.
  - **`START_FAILED reason=BUILD_FAILED`** → 컴파일/기동 오류. `bgt-be/build/bootrun.log`에서 원인을 확인한다. 수정 범위(`bgt-be`) 안이면 `bgt-be-implementer`로 고쳐 재기동, 밖(배포 SP·인프라)이면 BLOCKED(D-18 §5).
  - **`START_TIMEOUT`**(로그에 실패 패턴이 없는데 300초 내 미기동) → gradle 데몬 로그(`%USERPROFILE%\.gradle\daemon\{버전}\daemon-{pid}.out.log`) 또는 `bootrun.log`에서 원인을 확인한다.
- 스폰은 WMI(`Win32_Process.Create`)를 쓴다 — 샌드박스 셸의 `Start-Process` 자식은 잡 오브젝트로 함께 종료된다(실측). `JAVA_HOME`은 시스템 환경에 없으므로 스크립트가 기동 명령에 직접 주입한다(실측).

### Phase 3: 병렬 구현

Phase 2.5 합의(및 BE 인계) 완료 후 세 에이전트를 동시에 시작한다. 같은 브랜치에서 실행한다. **FE 구현은 반드시 `bgt-fe-implementer`, BE 구현은 반드시 `bgt-be-implementer` 서브에이전트(agentType)로 디스패치한다**(QA는 일반 에이전트). 이후 단계의 "구현 수정"(검증 복귀 루프·Phase 7)에서 FE/BE를 재실행할 때도 동일하게 이 두 에이전트를 쓴다.

**구현 방식 (FE·BE 공통)** — 검증 전략은 사전 조건의 com 서버 가동 여부로 정한다.
- **com 서버 가동 중(기본)**: FE는 `yarn build:local` + `npx tsc --noEmit`로 컴파일을 검증하고, 기능 검증은 Phase 4의 Playwright 라이브 화면으로 수행한다(별도 TDD/vitest 불필요). BE는 `gradlew.bat test`(JUnit, DB 없이 standalone MockMvc) red-green 루프를 유지한다.
- **com 서버 미가동**: 라이브 검증이 불가하므로 FE도 `/superpowers:test-driven-development`(red-green-refactor)로 대체 검증한다. 테스트 러너(vitest)가 미설치면 먼저 설치한다(공개 npm 레지스트리). 구동 불가 시 그 사실을 보고하고 `yarn build:local` + `npx tsc --noEmit`로 최소 검증한다.

**FE 에이전트 (`bgt-fe-implementer`)** — `bgt-fe/src/docs/ko/Intro.md` 및 하위 문서를 먼저 읽는다.
- `bgt_path`에 화면 파일 생성. `callApi` 유틸로 API 호출.
- BGT 기존 패턴 우선, 없으면 `cst`에서 가져와 BGT에 맞게 수정.
- ASIS 기능 추가/삭제 금지.
- 사용자의 별도 요청이 없는 한 UI 구성은 asis와 최대한 비슷하게 구현한다. 버튼의 순서, 그리드의 순서 등
- **ASIS 그리드 헤더 중복 무시**: ASIS 화면에서 그리드 헤더가 두 줄로 반복 표시되더라도 무시하고, TOBE 그리드는 **한 줄 헤더**로 구현한다.
- 구현 시 ASIS 메뉴 path와 화면 코드를 화면의 대표 index.tsx 파일에 주석으로 표시한다.
- 먼저 ASIS 화면의 구성을 분석하고 위에서 부터 아래로 순차적 구현한다.
- 사용자의 주문이 없는 한 ASIS 화면과 TOBE 화면의 1:1 구성으로 구현한다.

**BE 에이전트 (`bgt-be-implementer`)** — `bgt-be/docs/Server.md`를 먼저 읽는다.
- Model → Repository → Service → Controller 순서로 구현.
- 프로시저 연동: `bgt-be/docs/sp`의 배포된 SP 패키지 문서를 먼저 확인한다. 해당 화면에 매칭되는 SP를 찾아 XML `CALL` 구문을 배포 시그니처와 정확히 일치시킨다(파라미터 순서·IN/OUT·커서). 임의로 인자를 추가하지 않는다(불일치 시 ORA-06550).
- resultMap/Model의 컬럼명은 SP 문서의 커서 컬럼 alias(예: `CODE`/`NAME`/`NAME_ENG`)와 정확히 맞춘다 — alias가 다르면 매핑이 null이 된다.
- `bgt-be/docs/sp`에 매칭되는 SP 문서가 없으면(SP 미배포) 임의 쿼리를 만들지 말되, **Controller·Service 영역까지는 구현**하고 응답은 **stub 데이터 약 30건**으로 넉넉히 반환한다(그리드·페이징 확인용). Repository/XML의 실제 SP `CALL` 연결은 `// TODO: SP 배포 후 연결` 주석으로 남겨 배포 후 연결한다.
- 구현 시 ASIS 메뉴 path와 화면 코드를 화면의 Controller 파일에 주석으로 표시한다.

**QA 에이전트** — FE/BE를 기다리지 않고 ASIS 분석으로 테스트 시나리오를 작성한다.
- 주요 기능 흐름, 예외 처리, 권한별 UI, ASIS 전체 기능 항목 포함.

### Phase 4: UI 레이아웃 검증 (.designer.cs 스펙 대조 + 비전 비교)

FE/BE 구현이 완료된 후, 개발 표준 검증(Phase 5)·QA 검증(Phase 6)에 들어가기 전에 UI 레이아웃이 제대로 구성되어 있는지 검증한다. 이 단계만 전담하는 **별도의 UI 검증 에이전트**를 새로 생성한다.

**UI 검증 에이전트** — 두 신호를 함께 써서 "빠진 건 없는가(구조)"와 "보기에 비슷한가(룩앤필)"를 분리해 검증한다. Playwright MCP가 로컬에 설치되어 있지 않으면 먼저 Playwright 플러그인을 설치한 뒤 진행한다.
- **렌더 전제 (Playwright 실검증)**: 사전 조건에서 확인한 com 호스트(dev `http://localhost:3000/com/`, bgt remote `:3009` vite dev, 접속 시 자동 로그인·미로그인 시 로그인 ID `sdcahc`)에서 Playwright로 실제 TO-BE 화면을 열어 UI 검증을 실행한다(별도 서버 기동·빌드 없이 진행). `browser_navigate`로 대상 화면 deep-link `http://localhost:3000/com/bgt/{module}/{screen-path}`(예: `…/com/bgt/cm/cntrct-ctrt/cntrct-ctrt-lst`)에 접속한 뒤, 아래 ①(.designer.cs 스펙 대조)·②(비전) 검증을 라이브 화면 대상으로 수행한다.
- **폴백**: com 호스트에 접속되지 않거나 렌더링에 실패하면 그 사실을 마이그레이션 보고서(검증 결과 요약)에 기록하고, 가능한 신호만으로(예: 소스 파싱 기반 ① 스펙) 폴백한 뒤 Phase 5로 넘긴다.

**① AS-IS UI 스펙 대조 (.designer.cs 기준, 결정적 신호 1순위)** — 렌더링 차이에 흔들리지 않고 필드 누락·구성 어긋남을 정확히 잡는다.
- AS-IS 스펙: Phase 0에서 확정한 screenid 폴더의 `*.designer.cs`(WinForms Designer 생성 파일)를 파싱해 컨트롤 목록·라벨·배치(그룹/탭·그리드 컬럼·버튼)를 AS-IS UI 스펙으로 추출한다.
- TO-BE 현재값: Playwright accessibility snapshot으로 라이브 TO-BE 화면의 실제 렌더 요소(라벨·폼 항목·그리드 컬럼·버튼)를 수집한다.
- AS-IS 스펙 ↔ TO-BE 수집을 대조해 누락/추가 항목·구성 어긋남 목록을 산출한다.

**② 비전 비교 (의미적 신호, 룩앤필)**
- `browser_snapshot`·`browser_take_screenshot`으로 TO-BE를 캡쳐해, `docs/migration/{화면명}/`의 ASIS 캡쳐(있으면)와 `*.designer.cs` 구성 대비로 비교한다. UI 요소 순서(위→아래·좌→우), 그리드 칼럼명, 깨진 레이아웃, 겹침·잘림, 검색 영역 정렬 규칙(같은 열 라벨 우측 끝 정렬·입력 너비 동일)을 ASIS·Phase 2 참고 화면 대비로 점검한다.
- **그리드 헤더 중복은 불일치 아님**: ASIS 화면에서 그리드 헤더가 두 줄로 반복돼 보이는 것은 TOBE 한 줄 헤더가 정상이므로 불일치로 잡지 않는다.

**검증 전략** — ①(.designer.cs 스펙 대조)로 필드·구성 완전성을 먼저 확정한 뒤 ②(비전)로 룩앤필을 점검한다. `.designer.cs`를 못 찾으면(스크린샷만 가용 등) ②(비전) 단독으로 폴백하고 그 사실을 보고서에 기록한다.

**도구** — 설치가 필요한 것은 로컬에 없으면 설치 후 사용하고, 구동 불가 시 그 사실을 보고서에 기록한다.
- **.designer.cs 파싱**: Phase 0에서 찾은 screenid 폴더의 WinForms Designer 파일. AS-IS UI 스펙(①)의 출처.
- **Playwright**: com 호스트에 뜬 TO-BE 화면의 accessibility snapshot·스크린샷 추출(MCP). ①의 TO-BE 수집과 ②의 비전 캡쳐에 쓴다. 로컬 미설치면 플러그인 설치 후 진행.

- ①·② 검증 결과 중 ASIS와 다른 항목은 아래 "검증 불일치 항목 반환 형식"(요소/기대값/현재값)으로 정리해 반환한다.
- 미비한 UI를 발견하면 해당 UI를 직접 다시 수정한다. 수정 후 재검증하여 해결을 확인한다.
- 누락된 기능 단위의 UI를 보강해야 하는 경우 `/migrate-feature-asis-to-bgt` 스킬을 사용해 이식한다.
- 검증 통과 또는 더 이상 자동 수정이 불가능한 상태가 될 때까지 검증→수정→재검증을 반복한 뒤, 결과를 정리해 Phase 5(개발 표준 검증)로 넘긴다.

### Phase 5: 개발 표준 검증 (.claude/rules/develop 기준)

UI 레이아웃 검증 완료 후, QA 검증(Phase 6)에 들어가기 전에 구현이 **BGT 개발 표준**(`.claude/rules/develop`)을 따랐는지 검증한다. 이 단계만 전담하는 **별도의 개발 표준 검증 에이전트**를 새로 생성한다.

**개발 표준 검증 에이전트** — `.claude/rules/develop/README.md`의 목차를 진입점으로, 이번 화면에 해당하는 케이스 문서를 매칭하고 그 문서의 표준(참고 원본·레시피·"흔한 실패와 가드"·검증 방법)을 기준으로 구현을 점검한다.
- 화면 유형(단일 그리드/마스터-디테일/팝업/폼/탭/멀티 그리드)에 맞는 `a-archetype` 문서와, 사용된 기능(그리드 CUD 저장, 코드 드롭다운, 비동기 로드, 파일 첨부, 권한, 모달, API)에 맞는 `b-feature`·`c-api` 문서를 모두 매칭한다.
- 각 케이스 문서의 "흔한 실패와 가드"를 체크리스트로 삼아 점검한다. 예: 저장 시 삭제행 포함 수집(`getGridJsonData`)과 `flagCd` 매핑, 그리드 1회 로드(`isInit` 토글)+`resetGrid`, `BusinessCode.BGT`(템플릿의 CST 금지), 조회 조건 변경 시 그리드 초기화(`useFormEffect(onReset)`)와 초기화 버튼(`onResetCondition`), SP CALL 인자와 배포 SP 시그니처 일치 등.
- 표준과 다른 항목은 아래 "검증 불일치 항목 반환 형식"(요소/기대값/현재값)으로 정리해 반환한다.
- 표준 위반을 발견하면 해당 코드를 직접 수정한다.
- 코드 검증은 각 케이스 문서의 "검증 방법"(`yarn build:local` + `npx tsc --noEmit`, 필요 시 `gradlew.bat test`)으로 확인한다.

**검증 실패 시 복귀 규칙 (UI·개발 표준·QA 공통)**

세 검증 단계 — Phase 4(UI 레이아웃), Phase 5(개발 표준), Phase 6(QA) — 중 **어느 하나라도 검증에 실패하면, 수정이 필요한 항목을 Phase 3의 FE/BE 구현 에이전트에게 위임해 고친 뒤(전체 재구현이 아니라 불일치 항목만 수정), 반드시 Phase 4 → Phase 5 → Phase 6 순으로 세 검증을 다시 수행**한다. 수정이 UI를 다시 틀어놓을 수 있으므로 Phase 4(UI)부터 재검증한다. 세 단계가 모두 통과할 때까지 이 `구현 수정 → Phase 4 → 5 → 6` 루프를 반복한다.

**반복 상한**: 이 루프는 **최대 10회**까지만 반복한다. 10회 안에 세 단계가 모두 통과하지 못하거나 남은 불일치가 자동 수정 불가로 판단되면, 반복을 멈추고 미해결 항목을 정리해 Phase 7(이슈 처리)로 넘겨 사용자에게 확인한다.

**검증 불일치 항목 반환 형식 (UI·개발 표준·QA 공통)**

세 검증 단계에서 검증 결과가 기준(Phase 4·6은 ASIS, Phase 5는 개발 표준)과 **다른 항목**은 아래 표 형식으로 반환해 재구현·수정의 근거로 삼는다. 기준과 일치하는 항목은 나열하지 않는다.

| 요소 | 기대값 | 현재값 |
|------|--------|--------|
| {불일치가 발생한 화면 요소·폼 항목·그리드 컬럼·코드 위치} | {ASIS 또는 개발 표준 기준의 값·동작} | {현재 TOBE(BGT) 구현의 값·동작} |

- **요소**: 불일치가 난 지점(UI 요소·폼 항목·그리드 컬럼·`파일:라인` 등)
- **기대값**: ASIS 화면 또는 개발 표준(`.claude/rules/develop`) 기준으로 있어야 하는 값·동작
- **현재값**: 현재 TOBE 구현이 보이는 값·동작
- 이 표는 검증→수정→재검증 루프의 수정 대상 목록으로 쓰고, 최종 마이그레이션 보고서(§3)에도 반영한다.

### Phase 6: QA 검증

개발 표준 검증 통과 후 QA 에이전트가 Phase 3의 시나리오를 기준으로 구현 결과를 검증한다. ASIS와 다른 항목은 위 "검증 불일치 항목 반환 형식"(요소/기대값/현재값)으로 정리해 반환한다. 검증에 실패하면 위 "검증 실패 시 복귀 규칙"에 따라 Phase 3에서 불일치 항목을 수정한 뒤 Phase 4(UI)부터 재검증한다.

### Phase 7: 이슈 처리 (무개입)

실패 항목(반복 상한 도달분 포함)은 사용자에게 묻지 않는다 — 자동 수정 가능 항목은 상한 내에서 FE/BE 에이전트로 수정 후 "검증 실패 시 복귀 규칙"에 따라 Phase 4(UI)부터 재검증하고, 자동 수정 불가·범위 밖 항목은 BLOCKED로 정리해 보고서의 "미해결/사용자 조치" 절에 표(요소/기대값/현재값/사유)로 남긴다.

### Phase 8: 마이그레이션 보고서 생성 (최종)

모든 Phase(분석·구현·UI/개발표준/QA 검증·이슈 처리)가 끝난 뒤, 최종 결과를 종합해 마이그레이션 보고서를 작성한다.
- 저장 위치: 프로젝트 루트 `docs/migration/{화면명}/` 폴더 (없으면 생성). 이 폴더에는 Phase 1의 분석 보고서와 본 마이그레이션 보고서가 함께 저장된다.
- 파일명: `{화면명}_MIGRATION_REPORT.md`
- 형식: [MIGRATION_REPORT_TEMPLATE.md](MIGRATION_REPORT_TEMPLATE.md) 참고.
- 마이그레이션 개요(ASIS→TOBE 매핑), 구현 범위·생성/변경 파일 목록, 검증 결과 요약(Phase 4 UI·Phase 5 개발 표준·Phase 6 QA)과 분석 문서 참조, 그리고 테스트 시나리오 결과·SP 연동 및 컬럼 매핑 확인 결과·ASIS 대비 차이점·후속 작업을 포함한다.
- **검증용 임시 파일 정리 (스킬 종료)**: 스킬 실행 중 검증을 위해 생성한 임시 파일(FE TDD 대체용 vitest 테스트, 스크래치 검증 스크립트, Playwright 임시 산출물 등)은 보고서 작성 후 **모두 삭제**한다. 단, `docs/migration/{화면명}/` 산출물(분석 문서·마이그레이션 보고서·`_progress.md`·ASIS 기준 스크린샷)은 삭제하지 않는다.
