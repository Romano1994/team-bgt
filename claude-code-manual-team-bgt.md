# Claude Code 실전 활용 매뉴얼 — team-bgt 판

> 대상: BGT 프로젝트에서 Claude Code를 쓰는 개발자
> 목표: 같은 도구로 더 빠르고 정확하게, 그리고 **team-bgt 플러그인의 표준 워크플로우대로** 일하기

이 문서는 일반 Claude Code 매뉴얼(1~5부)에 **team-bgt 팀 전용 내용(6부)** 을 더한 판입니다.
Claude Code가 처음이면 1~5부부터, 이미 익숙하면 [6부. team-bgt 플러그인](#6부-team-bgt-플러그인)부터 보세요.

# 1부. 원리 이해

## 2. 4가지 습관

효율은 기능보다 습관에서 나옵니다.

1. **구체적으로 지시한다** — "로그인 고쳐줘"보다 "`auth.ts`의 토큰 만료 처리 버그를 고쳐줘"가 훨씬 빠릅니다.
2. **탐색 → 계획 → 구현 순서로 간다** — 큰 작업은 바로 코딩시키지 말고 먼저 코드베이스를 살펴보고 계획을 세우게 합니다.
3. **검증 수단을 준다** — 테스트, 실행 명령, 예상 결과를 알려주면 스스로 확인하며 정확도가 올라갑니다.

# 2부. 매일 쓰는 핵심기

## 3. 프롬프트 잘 쓰기

좋은 요청의 공통점은 **맥락**과 **목표**가 분명하다는 것입니다.

- 파일·폴더를 `@` 로 직접 참조하면 정확히 그 부분을 봅니다. (예: `@bgt-fe/src/pages/contract`)
- 이미지(스크린샷, 디자인)를 붙여넣어 UI 버그나 목업을 전달할 수 있습니다. ASIS 화면 캡쳐를 붙이면 이관 정확도가 올라갑니다.
- 여러 요구가 있으면 나눠서 순서대로 말하는 편이 낫습니다.

나쁜 예: "이거 좀 개선해줘."
좋은 예: "`CntrctBidList` 화면을 페이지네이션으로 바꿔줘. 페이지당 20개, 기존 API 응답 형식 유지."

## 4. 컨텍스트 관리

항상 남은 컨텍스트의 양이 60% 이상을 유지하도록 합니다.
컨텍스트가 차면 토큰 소비가 많아지고 답변의 품질이 확연히 떨어집니다.

- `/context` — 현재 컨텍스트 사용량을 확인합니다.
- `/compact` — 대화를 요약해 공간을 확보합니다. (핵심은 유지)
- `/clear` — 대화를 초기화합니다. 새 작업을 시작할 때 좋습니다.
- 대용량 탐색·조사는 **서브에이전트**에 맡겨 메인 대화를 가볍게 유지합니다. (7장) team-bgt에서는 `codebase-locator`(위치 조회), `bgt-be-implementer`·`bgt-fe-implementer`(무거운 읽기를 격리 컨텍스트에 가둠)가 이 역할을 합니다.

팁: 작업 단위가 바뀌면 `/clear` 로 새로 시작하는 것이 긴 대화를 억지로 이어가는 것보다 빠릅니다. 마이그레이션처럼 긴 작업은 중간에 compaction이 일어나도 디스크 산출물(`docs/migration/{화면명}/_progress.md`)로 상태를 복원하도록 설계되어 있습니다([6부 마이그레이션](#63-핵심-워크플로우-asis--bgt-화면-마이그레이션) 참고).

### 컨텍스트는 "한 모듈"만 담게 쪼갠다

하나의 대화 컨텍스트에는 **한 모듈만** 올리는 것이 정확도가 높습니다. 화면 하나도 통째로 넣지 말고 단위로 나누세요.

- **리스트 화면 / 디테일 화면**을 각각 다른 컨텍스트로 분리.
- 리스트 화면 안에서도 **검색(조회 조건) 영역 / 그리드 영역**을 나눠서 작업.
- **같은 컨텍스트에서 비슷한 실수가 반복되면 `/clear`로 새 컨텍스트에서 다시 시작**하세요. 그 컨텍스트가 잘못된 정보 하나에 집착(고착)하고 있을 확률이 높습니다 — 대화를 억지로 설득하기보다 비우고 새로 지시하는 편이 빠릅니다.

> 마이그레이션 스킬이 화면을 목록/상세/공통팝업 영역으로 쪼개 서브에이전트에 나눠 맡기는 것도 같은 원리입니다([6.3 Phase 1](#63-핵심-워크플로우-asis--bgt-화면-마이그레이션)).

## 5. 플랜 모드 & 체크포인트

**플랜 모드**는 실제 수정 전에 "무엇을 할지" 계획을 먼저 보여주는 모드입니다. 복잡한 리팩터링·마이그레이션에서 특히 유용합니다. 계획을 검토·승인한 뒤 실행하므로 사고를 줄입니다. 기본값으로 설정할 수도 있습니다.

**체크포인트**는 작업 중간 상태를 자동으로 기록합니다. 결과가 마음에 안 들면 **rewind**로 이전 시점으로 되돌릴 수 있습니다. 단, Bash 명령이 만든 변경이나 외부 변경은 추적되지 않으니 버전 관리(Git)의 대체물은 아닙니다. → BGT에서는 커밋을 `commit` 스킬로 관리합니다([6.3](#64-핵심-워크플로우-커밋)).

# 4부. 확장하기

## 7. 서브에이전트로 위임·병렬화

서브에이전트는 별도의 컨텍스트를 가진 보조 에이전트입니다.

- **컨텍스트 격리**: 대량 조사·로그 분석을 맡겨 메인 대화를 깨끗하게 유지.
- **병렬 처리**: 여러 관점의 조사나 리뷰를 동시에.
- **전문화**: 코드 리뷰어, 디버거 등 역할별로 정의 가능.

team-bgt는 BGT 코드베이스에 특화된 서브에이전트 6종을 제공합니다 — 구현용 `bgt-fe-implementer`·`bgt-be-implementer`, 검증용 `bgt-fe-verifier`·`bgt-be-verifier`·`uiux-verifier`, 위치 조회용 `codebase-locator`. 무거운 작업을 이런 전용 에이전트에 넘기면 메인 대화의 컨텍스트를 아낄 수 있습니다. 자세한 역할·호출 시점은 [6.2 agents](#agents-6종)를 보세요.

## 8. Skills & Plugins로 반복 작업 자동화

**Plugins**는 스킬·서브에이전트·훅·MCP 서버·커맨드를 하나로 묶어 배포하는 단위입니다. 팀이 같은 설정을 공유할 때 유용합니다. **team-bgt** 자체가 바로 그런 플러그인입니다 — [6부](#6부-team-bgt-플러그인)에서 전부 다룹니다.

## 9. Hooks로 이벤트 자동화

훅은 특정 이벤트가 발생할 때 스크립트를 실행하는 기능입니다. 반복 규칙을 자동화할 수 있습니다.

자주 쓰는 예:

- 파일 편집 후 자동 포맷팅
- 보호된 파일 편집 차단
- 작업 완료 시 알림(소리·데스크톱)
- 컴팩션 후 컨텍스트 재주입

주요 이벤트에는 `PreToolUse`, `PostToolUse`, `SessionStart`, `Notification`, `Stop` 등이 있습니다. `/hooks` 로 현재 설정을 확인합니다. 훅은 임의 명령을 실행하므로 신뢰할 수 있는 스크립트만 등록하세요.

team-bgt는 두 개의 훅으로 안전장치를 겁니다: **Stop 훅**(대규모 fe 변경이 있으면 응답 직전 fe 검증 게이트를 자동 실행)과 **pre-commit 훅**(민감 파일 커밋을 하드 차단). [6.2 hooks](#hooks)를 보세요.

# 5부. 운영 팁

## 10. 비용·토큰 아끼기

- `/usage` 로 사용량과 한도를 확인합니다.
- **모델 선택은 토큰 효율에 직접 영향을 줍니다.** 작업 난이도에 맞춰 고르고 `/model` 로 전환합니다. 오타 수정·포맷팅·간단한 보일러플레이트엔 가벼운 Haiku, 일반 기능 개발엔 Sonnet, 복잡한 설계·대규모 리팩터링엔 Opus가 적합합니다(가벼울수록 싸고 빠르지만 덜 똑똑함). 각 모델의 컨텍스트·비용은 아래 [Claude Code 모델](#claude-code-모델)을 보세요.
  - **워크플로우별 배분(팀 기준)** — 어려운 단계엔 Opus, 넓고 값싼 단계엔 Sonnet을 둡니다.

    | 워크플로우 | 모델 배분 |
    |---|---|
    | 계획 → 구현 | 계획 **Opus** / 구현 **Sonnet** |
    | 탐색 → 구현 | 탐색 **Sonnet** / 구현 **Opus** |
    | 계획 → 탐색 → 계획 → 구현 | 계획 **Opus** / 탐색·구현 **Sonnet** |

  - team-bgt 마이그레이션은 단계별로 모델을 자동 배분합니다 — **분석·검증은 Opus**(깊고 엄격하게), **구현은 Sonnet**(표준대로 꼼꼼히). 이 배분은 스킬이 알아서 하므로 신경 쓰지 않아도 됩니다.
- **statusline으로 사용량을 실시간 확인합니다.** 예: `Opus 4.8 (1M context) | 5h: 6% used | reset in 4h 37m`
  - `Opus 4.8 (1M context)` — 현재 모델(과 컨텍스트 크기).
  - `5h: 6% used` — **5시간 rate limit(사용 한도) 창을 몇 % 소진했는지**입니다. 한도 대비 **백분율**이에요.
  - `reset in 4h 37m` — 그 5시간 창이 리셋되기까지 남은 시간.
- **컨텍스트를 작게** 유지합니다. `/compact`·`/clear`·서브에이전트 활용. team-bgt implementer/locator가 무거운 읽기를 격리 컨텍스트에 가두는 이유가 이것입니다.
- 장황한 규칙은 `CLAUDE.md` 대신 필요할 때만 로드되는 **스킬**로 옮깁니다.
- MCP 서버가 많으면 도구 설명이 컨텍스트를 잡아먹으니 필요한 것만 켭니다.

### Claude Code 모델

Claude Code에서 `/model` 로 고르는 모델은 크게 세 등급입니다. 가벼울수록 싸고 빠르며, 무거울수록 똑똑합니다.

| 모델 | 등급 | 컨텍스트 | 최대 출력 | 쓰기 좋은 곳 |
|---|---|---|---|---|
| **Haiku 4.5** | 가벼움·최저가·최속 | 200K | 64K | 오타·포맷·간단 보일러플레이트 |
| **Sonnet 4.6** | 균형 | 1M | 64K | 일반 기능 개발·탐색 |
| **Opus 4.8** | 최상위(기본) | 1M | 128K | 복잡한 설계·대규모 리팩터링·검증 |

- **현재 기본 모델은 Opus 4.8 (1M 컨텍스트)** 입니다.
- 더 위 등급으로 **Fable 5**(Anthropic의 가장 강력한 모델, 1M 컨텍스트)가 있습니다. Opus보다 비싸고 한 요청이 수 분씩 걸릴 수 있어, 가장 어려운 장기 에이전트·추론 작업에만 씁니다.
- **Fast 모드(`/fast`)** — Opus를 **더 빠른 출력**으로 돌립니다(작은 모델로 낮추는 게 아님). Opus 4.8/4.7/4.6에서 지원.
- 상대 비용(입력/출력, 100만 토큰당 · API 정가 기준): Haiku `$1/$5` · Sonnet `$3/$15` · Opus `$5/$25` · Fable 5 `$10/$50`. 구독에서는 토큰당 과금이 아니지만, **무거운 모델일수록 5시간·주간 사용 한도를 빨리 소진**합니다(§10 statusline 참고).

#### Fable 5 자세히

Anthropic의 **가장 강력한 모델**로, 이전 모델이 버거워하던 난도에서 진가를 냅니다.

- **잘하는 것** — 밤새 돌리는 장기 자율 실행, 잘 정의된 시스템의 한 방(one-shot) 구현, 깊은 코드 리뷰·디버깅, 흐릿하거나 조밀한 이미지 이해, 여러 서브에이전트 병렬 오케스트레이션.
- **긴 턴** — 어려운 작업은 한 요청이 몇 분씩 걸립니다. 빠른 인터랙티브 수정보다 **"맡겨두고 나중에 확인"**하는 방식에 맞습니다.
- **비용·한도** — Opus의 2배 정가(`$10/$50`)에 더해, 같은 내용도 토큰이 **약 30% 더** 잡히는 토크나이저를 씁니다. 그만큼 한도가 빨리 닳으니 정말 어려운 작업에만.
- **안 맞는 곳** — 일상 수정·간단 조회·대량 반복 작업은 Sonnet/Opus가 더 빠르고 값쌉니다. "가장 센 모델"이 항상 정답은 아닙니다.

## 11. 메모리 관리

- **`/memory`** — 위 CLAUDE.md와 자동 메모리 파일을 보고 편집하는 명령.
- **파일 기반 자동 메모리** — `memory/` 디렉터리에 사실 하나당 `.md` 파일 하나로 쌓고, `MEMORY.md`가 그 인덱스입니다.

### 메모리가 컨텍스트·토큰을 얼마나 먹나

- **CLAUDE.md와 `MEMORY.md` 인덱스는 세션마다 컨텍스트에 로드**되어, 매 턴 대화와 같은 예산을 나눠 씁니다. 즉 이 둘이 커질수록 **실제 작업에 쓸 컨텍스트가 줄고 토큰 비용이 늘어납니다.**
- 인덱스는 상한을 넘으면 잘립니다(문서 기준 약 200줄/25KB). 그래서 **한 메모리당 한 줄**로 유지합니다.
- 반면 **개별 사실 파일(`memory/*.md`)은 시작 시 자동 로드되지 않고**, 관련될 때만 필요에 따라 읽힙니다. → 자세한 내용은 개별 파일에 두고 인덱스는 짧게.

### 효율적으로 유지하는 팁

- CLAUDE.md는 **짧고 명령형**으로(200줄 이하 목표). 길고 모호하면 오히려 잘 안 지켜집니다.
- 장황한 절차·규칙은 메모리가 아니라 **필요할 때만 로드되는 스킬**로 옮깁니다(시작 시 컨텍스트를 안 먹음). team-bgt가 규칙을 스킬로 배포하는 이유입니다.
- **틀렸거나 낡은 메모리는 지웁니다.** 남겨두면 컨텍스트만 먹고 잘못된 정보에 고착시킵니다.

# 6부. team-bgt 플러그인

여기서부터가 BGT 팀 전용 내용입니다.

## 6.1 team-bgt란

**team-bgt**는 BGT 팀 공용 Claude Code 플러그인입니다. BGT 코드베이스(`bgt-fe`/`bgt-be`)와 ASIS 시스템(PMS/TEMS), 참고 프로젝트(cst)를 오가며 **화면을 이관하고, 구현하고, 표준으로 검증하고, 커밋하는** 우리 팀의 개발 흐름을 스킬·서브에이전트·훅으로 묶어둔 것입니다.

**함께 설치되는 의존 플러그인**: `superpowers`(브레인스토밍·TDD·플랜 등 프로세스 스킬), `ponytail`(과잉 설계 방지 리뷰). ponytail은 서드파티 마켓이라 팀 배포 전 마켓 등록이 선행돼야 합니다.

### 설치

```bash
# 마켓플레이스 배포 후 (팀원)
/plugin marketplace add DietrichGebert/ponytail
/plugin marketplace add Romano1994/team-bgt
/plugin install team-bgt@team-bgt

```

설치 후 최초 1회 편의 설정:

- `/setup-cc` — 터미널에서 `cc` 만 쳐도 `claude` 가 뜨도록 PowerShell 프로필에 별칭 설치(멱등, 유저 scope, Windows/PowerShell 전용).
- `commit` 하드 차단 훅 설치(선택) — `skills/commit/hooks/pre-commit` 내용을 `bgt-fe\.git\hooks\pre-commit`, `bgt-be\.git\hooks\pre-commit` 에 복사하면 수동 커밋까지 민감 파일을 차단합니다.

## 6.2 무엇이 들어있나

### agents (6종)

서브에이전트는 이름을 직접 부르거나(예: "codebase-locator로 위치 찾아줘"), 스킬이 내부적으로 호출합니다. 실무에서는 대부분 스킬이 알아서 부릅니다.

| 에이전트 | 역할 | 언제 호출되나 |
|---|---|---|
| `bgt-fe-implementer` | FE 구현(Write/Edit). template/cst 원본 대량 읽기를 격리 컨텍스트에 가두고 메인엔 요약만 반환 | 신규 화면 빌드·ASIS 이관 등 다중파일 FE 작업 |
| `bgt-be-implementer` | BE 구현(Write/Edit). docs/sp·기존 슬라이스를 격리해 읽고 메인엔 **API 계약**(엔드포인트·VO camelCase 키·SP alias↔키 맵)만 반환 | 신규 슬라이스·ASIS 이관 등 다중파일 BE 작업 |
| `bgt-fe-verifier` | FE 변경을 표준·포맷·과잉설계 관점에서 검증(읽기 전용) | 턴 종료 시(응답 직전) 또는 "프론트 검증해줘" |
| `bgt-be-verifier` | BE 변경을 표준·과잉설계 관점에서 검증(읽기 전용) | 턴 종료 시 또는 "백엔드 검증해줘" |
| `uiux-verifier` | `.claude/rules/UIUX/` 하우스 규칙과 정적 대조(읽기 전용, 렌더 불필요) | UIUX 검사 요청 또는 마이그레이션 Phase 4 |
| `codebase-locator` | 편집 전 수정 대상 위치(파일·함수)와 cst 참고 원본을 조회(읽기 전용, 덤프 안 함) | 고칠 위치가 불확실할 때 편집 전에 |

**핵심 원리**: implementer는 "무거운 읽기를 가두고 요약만 반환", verifier는 "편집 도중이 아니라 턴 종료 시에만, 지정된 범위만" 검증합니다. 몇 줄짜리 위치가 명확한 수정은 메인이 직접 하고 이 에이전트들을 부르지 않습니다.

## 6.3 핵심 워크플로우: ASIS → BGT 화면 마이그레이션

가장 많이 쓰는 흐름입니다. `/migrate-asis-to-bgt` 스킬이 Phase 0~8을 오케스트레이션합니다.

**호출 예시:**

```
/migrate-asis-to-bgt PMS > 예산/내역 > 도급 관리 > 도급내역 집계표  bgt-fe/src/pages/contract
```

두 파라미터가 필요합니다:
- `asis_menu_path` — ASIS 메뉴 path 또는 화면 코드(예: `MPMC10001`)
- `bgt_path` — BGT 대상 경로(예: `bgt-fe/src/pages/contract`)

**꼭 알아둘 규칙:**

  - 사용자가 스킬 실행 전에 해야할 것들 명시 -

- **검증 실패 시 복귀 루프** — Phase 4·5·6 중 하나라도 실패하면, 불일치 항목만 Phase 3 구현 에이전트에게 위임해 고친 뒤 **Phase 4 → 5 → 6 순으로 다시** 검증합니다(UI가 다시 틀어질 수 있어 4부터). **최대 10회** 반복 후 남은 건 Phase 7로.
- **com 서버 전제** — dev `http://localhost:3000/com/`(bgt remote `:3009` vite). 가동 중이면 Phase 4를 Playwright 라이브 검증으로, 미가동이면 FE도 TDD로 대체 검증합니다.

**화면 전체가 아니라 기능 하나만** 기존 BGT 화면에 얹을 때는 `/migrate-feature-asis-to-bgt` 를 씁니다(migrate 후속).

## 6.4 핵심 워크플로우: 커밋

`/commit` 은 BGT의 표준 커밋 흐름입니다. 기본 대상은 **fe·be 둘 다**("fe만"/"be만"으로 좁힐 수 있음).

**순서: develop pull → ponytail-review → commit**

1. **develop 동기화(생략 불가)** — 변경 유무와 무관하게 **모든 대상 레포**에서 `git pull origin develop`. (커밋할 게 없는 레포도 pull은 함.)
2. **가드** — 스테이징 목록을 직접 검사:
   - **민감 파일**(`.env`, `*.properties/.pem/.key/.p12/.pfx/.jks/.keystore`, `id_rsa`) → **즉시 중단**, 절대 커밋 안 함.
   - **문서 파일**(`.md`/`.txt`) → 포함 여부를 사용자에게 확인.
3. **ponytail-review(필수)** — 스테이징 diff의 과잉 설계를 점검. `Lean already. Ship.` 이면 진행, 지적이 있으면 사용자에게 "그대로 / 수정 후" 확인.
4. **커밋** — fe/be 레포별로 다른 한국어 메세지. **푸시는 안 함**(명시 요청 시에만).

**메세지 컨벤션**: `<type>: <한국어 제목>` (scope 괄호 없는 평문 prefix)
- `feat:` 신규 화면·기능·모듈 추가 — 예: `feat: 도급입찰 목록(메인) 화면 추가`
- `fix:` 버그/매핑/컴파일 오류 수정 — 예: `fix: 도급입찰 상세 PQ 마감시간 컬럼 매핑 수정`
- 끝에 마침표 없음, 한국어 명사형 종결(추가/수정/구현/보완).

> 현재 브랜치가 `develop`/`main`/`master` 면 커밋 전에 사용자에게 확인합니다.

**신규 화면 개발 표준 흐름**: `/feature-finder`(중복 확인) → 플랜 모드로 계획 → implementer로 구현 → `/standard-verify`로 검증 → `/standard-refactor`로 수정 → `/commit`.
